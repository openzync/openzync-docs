services.worker package¶

ARQ worker package for OpenZync background tasks.

The worker process is started via:

python -m services.worker.worker

Configuration is loaded from environment variables through WorkerSettings. Task functions are registered in worker.HIGH_QUEUE_TASKS and worker.LOW_QUEUE_TASKS as they are implemented (Phase 1b+).

services.worker.prompt_renderer module¶

Prompt template resolution with auto-injected context.

All prompt templates live in the prompt_templates database table and are resolved at runtime via resolve_prompt_template_by_type(). The resolved template_text is returned as-is (plain text, no Jinja2) — context is assembled and injected by the caller via build_enrichment_prompt().

The .jinja2 files under prompts/ are the canonical source of truth for system-default prompts (Option A). They are read at signup time by PromptTemplateRepository.seed_default_prompts() and at import time by import_system_template(). The runtime resolution path (get_active_by_type) only queries org-scoped DB rows.

Usage (workers):

from services.worker.prompt_renderer import render_prompt

system_prompt, ctx = await render_prompt(

“fact_extraction”, org_id=org_id, episode_id=episode_id, session_id=session_id, db_session_factory=session_factory, return_context=True,

) prompt = build_enrichment_prompt(system_prompt, ctx)

async services.worker.prompt_renderer.render_prompt(prompt_type, *, template_text=None, org_id=None, episode_id=None, session_id=None, user_id=None, project_id=None, graph_backend=None, db_session_factory=None, return_context=False, **extra_context)[source]¶

Fetch and return a system prompt with auto-injected context.

When org_id and db_session_factory are provided, context variables are auto-fetched from the DB according to the registered TYPE_DATA_SOURCES for the given prompt_type. Explicit extra_context kwargs take precedence over auto-injected values.

When org_id is None (or db_session_factory is None), no auto-injection occurs — only extra_context and the optional template_text are used. This is the direct-invocation mode suitable for eval tests and one-off prompts.

The returned template_text is plain text (no Jinja2). Call build_enrichment_prompt() to assemble the full prompt.

Parameters:

  • prompt_type (str) – The template type (e.g. "fact_extraction"). Used to look up the registry of data sources to auto-inject.

  • template_text (str | None) – Raw template string. If None and org_id is provided, resolved from DB via resolve_prompt_template_by_type.

  • org_id (UUID | str | None) – Organisation UUID. Required for auto-injection.

  • episode_id (UUID | str | None) – Episode UUID. Used to fetch conversation content and recent history.

  • session_id (UUID | str | None) – Session UUID. Used to fetch known entities and existing facts for delta extraction.

  • user_id (UUID | str | None) – User UUID. Used to fetch user-specific data (episodes, facts, entities, classifications).

  • db_session_factory (async_sessionmaker[AsyncSession] | None) – Session factory for DB access. Required for auto-injection (and template resolution from DB).

  • return_context (bool) – If True, returns (prompt, context_dict) instead of just the prompt string. Useful for callers that need the injected context for post-processing (e.g. entity resolution after fact extraction).

  • **extra_context (Any) – Explicit template variables. These override any auto-injected values with the same key.

  • project_id (UUID | str | None)

  • graph_backend (GraphBackend | None)

Returns:

The system prompt text (plain text, no Jinja2). When return_context=True, returns (prompt, context_dict).

Raises:

  • ValueError – If template_text is None and cannot be resolved from the DB.

  • KeyError – If prompt_type is not in TYPE_DATA_SOURCES (only when auto-injection is active).

Return type:

str | tuple[str, dict[str, Any]]

Example

prompt = await render_prompt(
    "fact_extraction",
    org_id="...",
    episode_id="...",
    session_id="...",
    db_session_factory=session_factory,
)

services.worker.worker module¶

ARQ worker entrypoint — starts the worker pool, registers tasks, handles signals.

Usage:

python -m services.worker.worker

Environment variables are loaded via pydantic-settings from services.worker.worker_settings.WorkerSettings.

Architecture¶

Two separate ARQ worker pools run in a single process:

  • High-priority queue — real-time ingestion tasks (entity extraction, embedding, classification, graph sync).

  • Low-priority queue — batch / scheduled tasks (community summarisation, data ingestion, entity merge dedup).

Each pool has independent concurrency and timeout settings. The worker also exposes a Prometheus metrics endpoint and an aiohttp health-check server for Kubernetes liveness / readiness probes.

services.worker.worker.HIGH_QUEUE_TASKS: list[Callable[[...], Awaitable[Any]]] = [<function enrich_episode>, <function classify_dialog>, <function extract_entities>, <function embed_episode>, <function extract_facts>, <function embed_fact>, <function extract_structured>]¶

async services.worker.worker.main()[source]¶

Start the ARQ worker pool, Prometheus server, and health endpoint.

Startup sequence:

  1. Bootstrap from OpenBao — load secrets (DATABASE_URL, REDIS_URL) via worker-specific or main AppRole credentials (fail-fast).

  2. Configure structured logging

  3. Start the Prometheus metrics HTTP server on PROMETHEUS_PORT

  4. Create ARQ Redis connection settings from REDIS_URL

  5. Create high and low priority worker pools

  6. Register SIGTERM / SIGINT handlers for graceful shutdown

  7. Start aiohttp health check server on HEALTH_PORT (/health, /ready)

  8. Start queue depth monitoring as a background asyncio.Task

  9. Run both worker pools concurrently until a shutdown signal is received

Returns:

Never returns normally — always exits via signal handler.

Return type:

NoReturn

services.worker.worker_settings module¶

Worker-specific configuration — loaded from pydantic-settings.

All worker configuration lives here rather than in core.config because the worker process is independently deployable and should not depend on API-specific settings. Import the settings singleton rather than instantiating WorkerSettings directly.

Usage:

from services.worker.worker_settings import settings

arq_redis = await create_pool(RedisSettings.from_dsn(str(settings.REDIS_URL))) queue = settings.high_queue_full
